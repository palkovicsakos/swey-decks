How SWEY builds
AIOS capabilities

The spectrum, the pipeline, the discipline — a self-developing operating system that ships one capability at a time

The thesis

AIOS develops itself

  • 01
    Capabilities compound. Every skill, automation, and tool we ship also sharpens the machine that builds the next one.
  • 02
    The pipeline is the same at every tier. Skill, automation, tool — the shape of understand→build→harden never changes; only the depth scales.
  • 03
    Atlas spots. Build-AIOS builds. The operating system watches for repeated manual effort and routes it into the right tier, automatically.
The hard lesson
4×

Cost overrun on an internal tool built before the design was understood

Building without a locked design costs four times more than building after one. The fix is a heavy ideation front-half — measure twice, cut once. Code is the cheap part; clarity is scarce.

Part 01

Skill ⊂ automation ⊂ tool

The spectrum

One pipeline shape, three tiers

Tier 01
Skill
A prompt-driven capability packaged into a reusable SKILL.md. No code, no deploy. Runs inside Claude Code on demand.
› Light build loop
Tier 02
Automation
A script or worker that runs on a schedule or trigger. Deployed to VPS or systemd. Tested and cost-gated before launch.
› Medium build loop
Tier 03
Tool
A full product surface with a UI, VPS deploy, and its own lifecycle. Requires UI direction and GStack end-to-end.
› Full build loop

Half 2 · the builder

The autonomous loop

/office-hours → /autoplan
Plan
GStack self-driving: office-hours surfaces the approach, autoplan breaks it into atomic tasks. Decisions are batched — one structured ask, never a drip of questions.
Trigger: locked PRD in hand
/build → /review → /qa → /ship
Ship & harden
Build runs autonomously. Review and QA gate before ship. Post-ship retro encodes every corrected step as a hard gate in the skill file itself — the tool cannot repeat the mistake.

The discipline

Three rules that hold the system

  • 01
    The board is the "why build this" gate. Leverage or hiding place? Every capability must pass before build starts. No exception for urgent-feeling ideas.
  • 02
    Every capability is grounded in workspace context. Map existing state first. Never build on top of an unmapped system — it doubles the cost and creates invisible debt.
  • 03
    Built so Kitti runs it solo. If a capability requires Ákos to operate it, it hasn't been packaged. The definition of done includes a Kitti-runnable SOP.

The takeaway

Every capability we build also sharpens the machine that builds capabilities

Skill → automation → tool. The same pipeline, deepening with weight. Atlas spots. Build-AIOS ships. The retro hardens. Day by day, AIOS develops itself — and the cost of the next build drops while the output quality rises. This is not a project. It is a compounding asset.
